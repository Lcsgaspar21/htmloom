Why rename "Target Priority" inside the popover?

SurfaceLabel todayValue displayedWhat it means
Scorecard / My Network Priority — stays Mid (base, fixed) How important the supplier is to me, in general
Alert popover Target Priority — must change High (= base × alert impact) How impacted this supplier is by this specific alert

Same word, two different meanings, two different numbers — that's the linguistic root cause of REAL-415. Since the scorecard label stays, the popover label has to give. Candidates: Per-target severity · Severity per target · Target severity · How this alert affects each target. The exact word is up for grabs, but it cannot be "priority".

How are these severities calculated?
Each severity above combines the target's base priority (set in their profile) with how strongly this alert affects them. The result can be higher than the base when the alert hits hard, or lower when it barely touches the target.

Pros. Zero per-row visual noise. One discoverable element. Cheapest to ship.

Cons. Generic. The user still has to map "the explanation" back to the specific row that confused them. Fails the most acute case: when one target differs from the rest (e.g. Acme is Low while the others are High) — the footer gives no row-level signal.

Decisions to make consciously

Decision Recommendation Why
Section header rename Yes — "Target Priority" → "Per-target severity" Cheapest, highest-leverage change. Defuses REAL-415 even before the icons land. Non-negotiable.
Tooltip-in-popover, is it OK? Yes only via discoverable signifier (ⓘ icon). Never on the whole row. Standard practice across Atlassian, Linear, Stripe, Notion, Figma. The signifier is what makes it acceptable.
Per-row vs footer-only Per-row. Footer-only fails REAL-415's question shape. The user's question is "why is this Chevron High?", not "how does the formula work in general?". A per-row answer is exact; a footer answer requires mapping.
Icon on every row, or only on anomalies? Only on anomalies (base ≠ result). Becomes a signal, not noise. Reduces visual weight by ~80% in typical alerts (95% of impact scores are Mid).
Footer link to docs? Yes, single line. No side-sheet. Catches power users who want the whole formula. ~20px of footer space. Cheap to ship.
Tooltip trigger: hover or click? Both. Hover on desktop, tap on touch (standard tooltip primitive). Touch fallback is mandatory; never rely on hover alone.
Tooltip content style Short narrative: "Base Mid, alert impact High → High" ~120 chars. Fits in a small tooltip without scrolling. Reads in plain language.

Recommendation

Ship Layout A. Section header rename + reserved icon column + ⓘ on anomalous rows + footer link to docs. Dots stay aligned, signifier is discoverable, footprint is ~14px of right gutter that you'd probably want anyway for visual breathing room.

Layout B is the elegant fallback if the team rejects the extra gutter. Footer-only is the cheap fallback if engineering bandwidth is the constraint, but it doesn't fully solve REAL-415 because the user's question is per-row, not generic.

The section rename ships independently of which layout you choose — and is the single biggest defuse for REAL-415-class bugs even on its own.
